Netflix Monolithic to Microservices

Will Otterbein, COMP4537

Netflix's early architecture was a single, monolithic application. While simple to start, this design made it difficult to scale and update as the company grew. Every change required modifying the entire system, a process that became slow and complicated. To solve this, Netflix began transitioning to an AWS (Amazon Web Services) powered microservices architecture in 2008. This move allowed them to achieve the massive scalability, reliability, and speed they needed to become the streaming giant we know today.

Netflix Microservices (A few Examples)

This section attempts to explain some of the microservices and microservice related technologies that Netflix employs.

  • EC2 → Amazon scalable compute power. Accommodate variable user demand.
  • S3 → Amazon unstructured data storage.
  • RDS → Amazon relational databases for structured data storage.
  • Netflix Zuul → Netflix API gateway, receives incoming requests and routes them to the correct service.
  • Netflix Eureka → Allows other services to discover each other without having to “know” the service beforehand.
  • Authentication/Authorization API → Handles user authorization and authentication.
  • Search API → Queries Netflix content/data stores and returns results.
  • Discovery/Recommendation API → Powers the recommendations and discovery feature (algorithms powered by ML) to serve users content that they may also want to watch.
  • Async Messaging API → Facilitates asynchronous communication between services.
  • Open Connect CDN → Custom CDN Netflix built to optimize the delivery of video.

Netflix services are responsible for their own domain of data and therefore have their own databases. This is known as the database-per-service pattern. This ensures that Netflix microservices can function independently. Netflix enforces data consistency by syncing data between services over time.

Pros and Cons of Microservices

This section explores some of the pros and the cons of microservice architectural patterns.

Pros Cons
Scalability → System compute can be scaled in accordance with variable demand from users. Increased complexity → Managing the communication between services is a challenge.
Modularity → System can be modified at a granular level, as long as system API remains consistent. Deployment and Version Coordination → Coordinating the deployments and versions between services introduces additional considerations with respect to compatibility issues between services.
Fault Isolation → System faults are isolated to specific services, which makes the process of identifying and correcting the source of the fault less daunting Testing complexity → Specifically, integration testing. Orchestrating tests across multiple services is challenging and complex.
Quicker Deployments → The entire system does not need to be redeployed if only 1 system needs to be updated or is receiving changes. Data management → Data consistency across services can be complicated and can lead to discrepancies between data if not handled correctly.
